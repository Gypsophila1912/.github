## ブランチ戦略

```
main          本番相当。直接 push 禁止、タグを打つとリリースされる
develop       開発の集約ブランチ。PRのデフォルトマージ先
feature/xxx   新機能
fix/xxx       バグ修正
chore/xxx     リファクタ・設定変更・依存更新など
docs/xxx      ドキュメントのみの変更
```

**ブランチ名の例**

```
feature/add-login
fix/redirect-after-logout
chore/update-deps
docs/update-readme
```

## コミットメッセージ

[Conventional Commits](https://www.conventionalcommits.org/ja/) に準拠してください。

```
<type>(<scope>): <summary>
```

**type 一覧**

| type       | 使いどき                           |
| ---------- | ---------------------------------- |
| `feat`     | 新機能                             |
| `fix`      | バグ修正                           |
| `docs`     | ドキュメントのみ                   |
| `style`    | フォーマット変更（動作に影響なし） |
| `refactor` | リファクタリング                   |
| `test`     | テストの追加・修正                 |
| `chore`    | ビルド・CI・依存関係など           |
| `revert`   | コミットの取り消し                 |

**例**

```
feat(auth): ログイン後にダッシュボードへリダイレクト
fix(api): 空のレスポンスで500エラーになる問題を修正
chore(deps): eslint を v9 に更新
```

## プルリクエスト

1. `develop` から作業ブランチを切る
2. 作業が完了したら PR を `develop` に向けて作成
3. [PRテンプレート](..//pull_request_template.md) に沿って記入する
4. CI がすべてグリーンになっていることを確認
5. レビュワーをアサインする（CODEOWNERS で自動アサインされます）

**PRのサイズ感**

レビューしやすくするため、1PRあたりの変更は 400行以内を目安にしてください。
大きくなる場合はイシューで事前に設計を相談してから実装に入ることを推奨します。
